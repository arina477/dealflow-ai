# Wave 17 — P-3 Plan (multi-spec: M8 pilot-partner data-isolation, 4 blocks)

## Approach
### Action 1 — Architecture deltas
- **workspaces + workspace_id scoping [seed 0db154ff].** New `workspaces` table (id uuid pk, name, created_at) + a single default workspace (the pilot firm). `workspace_id` FK column on every tenant table (the 12 schema files: users, mandates, data_source_connections, audit_log_entries, outreach, pipeline, compliance/disclaimer, etc.). Existing rows BACKFILLED to the default workspace before `NOT NULL` cutover. **audit_log_entries.workspace_id added OUTSIDE HashableEntryFields** (the wave-14 mandate_id precedent) so the HMAC chain stays byte-identical (verifyChain ok:true). **[P-4 F4 recorded posture]:** workspace_id is hash-excluded so it is NOT chain-covered — a raw UPDATE of workspace_id would re-attribute a record cross-workspace without breaking verifyChain; ACCEPTED (as mandate_id) BECAUSE the WORM BEFORE-UPDATE trigger blocks it, and df2f3b2f MUST prove the trigger rejects a workspace_id UPDATE (isolation-integrity rests on the tested WORM trigger, not the HMAC). *Alt:* schema-per-tenant — REJECTED (heavy, wrong for one pilot firm; shared-table + workspace_id + RLS is the standard isolation primitive).
- **Deny-by-default RLS [e45ba68c].** Per tenant table: `ENABLE ROW LEVEL SECURITY` + **`FORCE ROW LEVEL SECURITY`** (the API runs as the table-OWNING role — without FORCE the owner bypasses RLS and every isolation test is a false-green — problem-framer A). Deny-by-default policy: `USING (workspace_id = current_setting('app.workspace_id', true)::uuid)`; unset GUC → NULL comparison → 0 rows (fail-closed; NO COALESCE-to-default, NO `= ''` fail-open shape). **audit_log + recordkeeping-export READ-PROJECTION are inside this boundary** (ceo-reviewer). *Alt:* app-layer WHERE-filtering — REJECTED (leaks the moment one query forgets the filter).
  - **[P-4 F2 — HIGH] Resolver carve-out:** the `users` identity/workspace-resolution lookup (getUserWithRole, run BEFORE the GUC is set) MUST be RLS-exempt — else chicken-and-egg (users read → 0 rows → workspace never resolves → every request fails closed). Implement as a `SECURITY DEFINER` function `resolve_user_workspace(st_user_id) RETURNS uuid` (RLS-bypassing, returns only the caller's own workspace_id + role) that runs FIRST; the GUC is set from its result; all subsequent `users` reads are then RLS-scoped normally. `users`/`roles`/`invites` stay FORCE-RLS for the post-GUC data path; only the bootstrap resolver bypasses.
  - **[P-4 F3 — MEDIUM] Chain integrity vs visibility split:** the audit HMAC chain is a SINGLE GLOBAL chain (genesis-anchored, monotonic global sequence, one advisory-lock key). RLS-scoping the audit READ two ways would break it: scoped-verify sees a non-contiguous subset → false `ok:false sequence-gap`; unscoped-out-of-tx verify → `{ok:true, entriesChecked:0}` vacuous false-green. **RESOLUTION: separate integrity from visibility.** (a) verifyChain's integrity WALK (`readChainAscending`) runs RLS-EXEMPT via a `SECURITY DEFINER` verifier function / dedicated verifier role — integrity is GLOBAL, so verifyChain stays `ok:true` over the whole chain (NOT per-workspace). (b) The LIST/EXPORT read PROJECTION (admin-activity, recordkeeping list + the exported entry set) IS workspace-scoped by RLS — a user sees/exports only their workspace's entries. Reconcile `recordkeeping.service.ts:210`'s out-of-tx verify to the RLS-exempt integrity path. Do NOT convert to per-workspace sub-chains (far larger change; out of scope).
- **Request-scope workspace propagation [96026365 — P-4 F1 CRITICAL rework].** The shipped tenant READS run on the POOLED `this.db` handle OUTSIDE any transaction (mandate/audit/recordkeeping reads — 41 modules); a tx-local `SET LOCAL` would make every non-tx read return 0 rows (app bricked) and push toward a session-level SET that leaks across pooled connections. **RESOLUTION: request-scoped dedicated connection.** A NestJS request-scoped provider checks ONE pg client out of the pool for the request lifetime, runs `SET app.workspace_id = $wsId` (session-level ON THAT DEDICATED CLIENT ONLY) after resolving `$wsId` from `resolve_user_workspace`, binds the request's Drizzle handle to that client so EVERY tenant query (reads INCLUDED) sees the GUC, and **`RESET app.workspace_id` (or `DISCARD ALL`) in a `finally` before the client returns to the pool** — so no workspace ever leaks to the next request on that connection. Background jobs / unauthenticated paths get NO GUC set → RLS denies (fail-closed). Greenfield (0 existing current_setting/SET LOCAL). *Alt considered:* wrap every read in `runInTransaction` + `SET LOCAL` — REJECTED (a 41-module refactor of every read path; the dedicated-connection-with-RESET is the standard RLS-over-pool pattern). *Alt:* naive session-level SET on the shared pool — REJECTED (the cross-tenant leak F1 names).
- **Cross-tenant negative-read test [df2f3b2f — + P-4 F1/F3/F4].** A real-DB e2e: 2 workspaces (A, B) each seeded; as a workspace-A user, reads of workspace-B rows return EMPTY across primary + audit_log LIST + recordkeeping-export; runs over the SAME request-scoped dedicated-connection path the API uses (false-green guard — problem-framer A). Fault-killing: removing FORCE / the policy / the RESET-in-finally / the propagation → cross-read → FAIL (assert exact-0 cross-tenant + non-zero same-tenant positive control). **[P-4 F4] Additionally assert the WORM trigger BLOCKS a `UPDATE audit_log_entries SET workspace_id=<other>` — proving the sole isolation-integrity control holds** (since workspace_id is hash-excluded, the WORM trigger is the only backstop against silent cross-tenant re-attribution; the test must prove it). **[P-4 F1] Assert no GUC leak: a second request on a reused pooled connection sees only its own workspace** (RESET-in-finally proof). WORM-safe teardown (T-4 rule 1).
