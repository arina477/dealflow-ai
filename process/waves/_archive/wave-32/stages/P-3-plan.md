### CORRECTED API-key provisioning (Action 1 item 3 SUPERSEDED — DB-seed was architecturally impossible):
karen (Twenty OSS source): API keys are **asymmetric-signed JWTs** (`generateApiKeyToken` signs with a private key from Twenty's jwtKeyManager DB tables, NOT APP_SECRET); the `apiKey` DB row carries NO presentable secret (auth = verify JWT signature + lookup row by jti). **A bare DB insert yields no usable token → DB-seed DROPPED.**
- **REAL headless key path (PRIMARY):** GraphQL `createApiKey` mutation → `generateApiKeyToken` mutation → capture the returned JWT as TWENTY_API_KEY. (Both are server resolvers reachable over HTTP with an authed session.)
- **The ACTUAL self-completability crux = WORKSPACE BOOTSTRAP** (the GraphQL key path needs an authenticated admin session). **SPIKE-FIRST, before any wiring**, one of:
  - (a) deploy with **`IS_SIGN_UP_ENABLED=true`** → programmatic GraphQL `signUp`→`signIn` against the fresh instance → session token → `createApiKey`+`generateApiKeyToken`; OR
  - (b) run a one-shot provisioning container at **`NODE_ENV=development`** for the first-class CLI **`workspace:generate-api-key`** (creates+signs, logs `TOKEN:<jwt>`) after a scripted signup.
- **Fallback if BOTH bootstrap spikes fail:** a ONE-TIME manual founder bootstrap (create the first admin via Twenty's UI once) → then everything downstream is headless. Escalate to head-builder to surface this single manual step (acceptable — it's a one-time instance-init, not per-run).
